    helm create — Scaffolding a Chart

    The helm create command generates a complete chart directory with best-practice defaults you can modify.

    What it scaffolds

    Run:

    helm create mychart

    Helm produces: Chart.yaml (metadata), values.yaml (defaults), charts/ (dependency tarballs), templates/ (Go template manifests), templates/NOTES.txt (post-install message), templates/_helpers.tpl (reusable named templates), templates/deployment.yaml, service.yaml, serviceaccount.yaml, hpa.yaml, ingress.yaml, templates/tests/test-connection.yaml, and .helmignore.

    Directory walkthrough

    Chart.yaml is the identity card — name, version, dependencies. values.yaml holds every configurable knob with sensible defaults. templates/ contains Go templates that Helm renders into K8s manifests. Files starting with _ (like _helpers.tpl) are partials, never rendered directly. The charts/ directory stores downloaded dependency charts. The optional crds/ directory holds CRDs.

    Key behaviors

    If the target directory doesn't exist, Helm creates it. You can use a starter chart with --starter to scaffold from a custom template instead of the built-in default. The default scaffold targets nginx — you'll replace the image, ports, and probe configuration for your own app.

    Gotcha

    The scaffolded templates are opinionated and verbose. Many teams strip them down to only what they need rather than trying to understand every conditional. Start by reading the generated _helpers.tpl — it defines the naming and labeling conventions used throughout all templates.

    Chart.yaml Deep Dive

    Chart.yaml is the chart's identity file with required and optional fields controlling versioning, dependencies, and compatibility.

    Required fields

    Three mandatory fields:

    apiVersion: v2
name: mychart
version: 1.2.3

    apiVersion: v2 is required for Helm 3. name must match the directory name. version must follow SemVer 2.0.0. Helm rejects a chart without them.

    Important optional fields

    type: "application" (default, installable) or "library" (not installable, provides templates). appVersion: the deployed app version — purely informational. kubeVersion: SemVer constraint on K8s version, e.g., ">= 1.27.0". deprecated: boolean marking the chart as deprecated.

    Metadata fields

    description: one-line summary for helm search. keywords: list for discovery. home/sources: project URLs. icon: URL to image. maintainers: list with name, email, url. annotations: freeform key-value pairs. As of Helm 3.3.2, unlisted fields are rejected.

    Dependencies block

    Listed under dependencies: as an array:

    dependencies:
  - name: postgresql
    version: "~12.1.0"
    repository: "oci://registry-1.docker.io/bitnamicharts"
    condition: postgresql.enabled
    alias: db

    Optional fields: condition (boolean path), tags (group toggling), alias (rename), import-values.

    helm dependency update

    This downloads into charts/ and generates Chart.lock.

    Gotcha

    appVersion is a string, not SemVer — it can be any value like a commit SHA. Don't confuse it with version, which must be SemVer and is what Helm uses for chart comparisons. Bumping appVersion without bumping version means the chart package is identical to Helm.

    Writing Templates — Go Template Basics

    Helm templates use Go's text/template language with {{ }} delimiters, pipelines, and whitespace control.

    Actions

    Everything between {{ and }} is an action. Plain text outside passes through unchanged. Actions can output values, call functions, or control flow:

    {{ .Values.name }}
{{ quote .Values.name }}
{{ if .Values.enabled }}...{{ end }}
{{/* comment */}}

    Pipelines

    Pipelines chain functions with |, passing the previous result as the last argument:

    {{ .Values.name | default "app" | quote }}

    Pipelines are the idiomatic way to compose functions in Helm templates.

    Whitespace control

    {{- trims all whitespace to the left. -}} trims to the right. There must be a space between the dash and the directive. This is critical because the template engine preserves all whitespace outside actions — without trimming, you get blank lines and broken YAML indentation.

    Common patterns

    Conditionally include a block:

    {{- if .Values.tolerations }}
tolerations:
{{ toYaml .Values.tolerations | indent 2 }}
{{- end }}

    Include a named template with proper indentation:

    {{ include "mychart.labels" . | nindent 4 }}

    Use nindent (not indent) when the include is inline after a key.

    Gotcha

    Aggressive {{- can eat too much whitespace and merge lines. A common bug: {{- end -}} everywhere collapses YAML structure. Be surgical — only trim the side with unwanted whitespace. Use helm template to preview output and verify indentation.

  • 4
    Built-in Objects

    Helm provides six top-level objects in every template: Release, Values, Chart, Capabilities, Files, and Template.

    Release

    Release object properties:

    .Release.Name
.Release.Namespace
.Release.IsInstall
.Release.IsUpgrade
.Release.Revision
.Release.Service

    .Release.Service is always "Helm". Use Release.Name for resource naming.

    Values

    .Values contains the merged result of defaults, parent overrides, -f files, and --set flags. Access nested values with dot notation:

    {{ .Values.image.repository }}

    Always use default or with blocks to guard against nil values for optional nested keys.

    Chart

    .Chart exposes Chart.yaml fields:

    {{ .Chart.Name }}
{{ .Chart.Version }}
{{ .Chart.AppVersion }}

    Fields are title-cased when accessed — "name" becomes .Chart.Name. You cannot modify Chart values in templates.

    Capabilities and Template

    Check cluster capabilities:

    {{ .Capabilities.KubeVersion.Version }}
{{ .Capabilities.APIVersions.Has "batch/v1" }}
{{ .Template.Name }}
{{ .Template.BasePath }}

    .Template.Name gives the full path like "mychart/templates/deployment.yaml".

    Files

    Access non-template files:

    {{ .Files.Get "config.ini" }}
{{ .Files.Glob "conf/*.toml" }}
{{ (.Files.Glob "conf/*").AsConfig }}
{{ (.Files.Glob "conf/*").AsSecrets }}

    .Files.Lines provides line-by-line iteration. Cannot access files in templates/ or excluded by .helmignore.

    Essential Template Functions

    Helm provides 60+ functions from Go templates and the Sprig library for string manipulation, data conversion, and validation.

    String and formatting

    quote / squote — wrap in quotes. upper / lower — case conversion. replace — substitution. trim / trimPrefix / trimSuffix. contains / hasPrefix / hasSuffix — boolean tests.

    {{ printf "%s-%s" .Release.Name .Chart.Name }}

    YAML and indentation

    toYaml converts any object to YAML string. Almost always piped to indent or nindent:

    {{ .Values.resources | toYaml | nindent 6 }}

    nindent prepends a newline then indents — use when inline after a key. indent adds spaces without a leading newline. toJson / fromJson / fromYaml for format conversion.

    Defaults and validation

    Fallback values, required fields, and error handling:

    {{ .Values.port | default 8080 }}
{{ required "image.repository is required" .Values.image.repository }}
{{ coalesce .Values.a .Values.b "fallback" }}
{{ fail "something went wrong" }}

    Advanced functions

    Template evaluation and conditional logic:

    {{ tpl .Values.myTemplate . }}
{{ lookup "v1" "ConfigMap" "ns" "name" }}
{{ ternary "yes" "no" .Values.enabled }}
{{ include "mychart.labels" . }}

    tpl evaluates a string as a template. lookup queries live cluster resources (empty with helm template).

    Data structure functions

    list — create a list. dict — create a dictionary. get / set — dictionary access. merge — combine dicts. keys / values — extract from dict. pick / omit — select or exclude keys. hasKey — check existence. range and index — iterate and access. Essential for building dynamic configuration blocks.

    Flow Control — if, with, range

    Helm's three control structures generate conditional content, modify scope, and iterate over collections.

    if/else

    Conditional blocks:

    {{ if PIPELINE }}...{{ else if PIPELINE }}...{{ else }}...{{ end }}

    A pipeline is false for: boolean false, 0, empty string, nil, or empty collection. Operators: eq, ne, lt, gt, le, ge, and, or, not.

    {{ if and .Values.ingress.enabled (eq .Values.ingress.className "nginx") }}

    with (scope modifier)

    Sets dot (.) to a new scope:

    {{ with .Values.favorite }}
  drink: {{ .drink }}
{{ end }}

    Critical: you cannot access outer objects with dot — use $ for root: {{ $.Release.Name }}. If the pipeline is empty/nil, the block is skipped (acts like an if).

    range (loops)

    Iterate over lists and maps:

    {{ range .Values.servers }}...{{ end }}
{{ range $key, $val := .Values.labels }}...{{ end }}
{{ range tuple "small" "medium" "large" }}...{{ end }}

    Like with, range changes dot scope — use $ for root.

    Scope and the dollar sign

    The dot (.) is the current scope — root at top level, changes inside with and range. The $ always points to root and never changes:

    {{ range .Values.items }}
name: {{ $.Release.Name }}-{{ . }}
{{ end }}

    This is the most common source of confusion in Helm templates.

    Gotcha

    Empty lines from if/end blocks pollute YAML. Always use {{- if ... }} and {{- end }} to trim whitespace. Another trap: {{ if .Values.foo }} is true for ANY non-empty value including "false". To check a boolean explicitly, use {{ if eq .Values.foo true }}.

    Named Templates — define, template, include

    Named templates are reusable blocks defined once and invoked anywhere across the chart.

    Defining templates

    Create named templates in _helpers.tpl:

    {{/* Generate standard labels */}}
{{ define "mychart.labels" }}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{ end }}

    You can split helpers across multiple _ files.

    template vs include

    template inserts output directly — cannot be piped. include returns a string — CAN be piped:

    {{ template "mychart.labels" . }}
{{ include "mychart.labels" . | nindent 4 }}

    Always prefer include over template for indentation control.

    Context passing

    The second argument sets the scope inside the named template:

    {{ include "mychart.labels" . }}
{{ include "mychart.labels" .Values.app }}

    The first passes full root context. The second passes only that subtree. Forgetting to pass context makes dot nil, causing nil pointer errors.

    Naming conventions

    Always prefix with the chart name: "mychart.fullname", "mychart.labels". Template names are globally unique across parent and subcharts — without the prefix, name collisions cause silent overwriting (last loaded wins).

    _helpers.tpl standard patterns

    The scaffold defines: mychart.name (chart name, truncated to 63 chars), mychart.fullname (release-name-chart-name), mychart.chart (chart-name-version), mychart.labels (standard app.kubernetes.io/* labels), mychart.selectorLabels (immutable selectors), mychart.serviceAccountName (conditional name). These form the foundation every template references.

    values.yaml Design

    Well-designed values.yaml makes your chart intuitive to configure with clear naming, documentation, and safe defaults.

    Naming conventions

    Start with lowercase. Use camelCase: replicaCount, servicePort. Never use hyphens — invalid in Go templates. Never start with uppercase — conflicts with Helm built-ins. These conventions are enforced by community standards and Artifact Hub.

    Flat vs nested

    Prefer flat when possible — serverName and serverPort is simpler than nesting under server:. Flat values eliminate nil checks. Use nesting only for many related variables where at least one is non-optional, like image.repository and image.tag.

    Documenting values

    Document every property with a comment. This enables documentation generators like helm-docs to auto-generate README tables. Keep descriptions to one line. Always include the type implicitly through the default value.

    Type safety

    Quote string values to prevent YAML coercion: env: "true" vs env: true (boolean). In templates, use quote to ensure strings stay strings:

    {{ .Values.name | quote }}

    Port numbers used in env vars should be stored as strings and converted with {{ int }}.

    Override-friendly design

    Prefer maps over lists for values users override — maps work with --set (servers.foo.port=80) while lists require error-prone index notation. Provide empty map {} as default instead of null. Keep values.yaml as a dev-friendly baseline with minimal resources and single replicas.

    NOTES.txt — Post-Install Instructions

    templates/NOTES.txt prints user-facing instructions after helm install or upgrade.

    How it works

    NOTES.txt in templates/ is processed like any template — all built-in objects and functions are available. The rendered output is displayed after install and upgrade but is NOT sent to Kubernetes. It's purely informational CLI output.

    What to include

    How to access the app (kubectl port-forward, ingress URLs), how to get generated credentials (kubectl get secret), how to check status. Include conditional instructions based on service type and ingress configuration.

    Example

    A conditional block:

    {{- if .Values.ingress.enabled }}
Visit http://{{ .Values.ingress.host }}
{{- else }}
Run:
  kubectl port-forward svc/{{ include "mychart.fullname" . }} 8080:{{ .Values.service.port }}
Then visit http://127.0.0.1:8080
{{- end }}

    Gotcha

    NOTES.txt is validated by helm lint — template errors block linting and installation. Since it runs through the template engine, a broken NOTES.txt can block an otherwise valid install. Test with helm template.

    Template Validation — lint, template, dry-run

    Three complementary commands catch progressively deeper issues: lint for structure, template for rendering, dry-run for API validation.

    helm lint

    Static analysis on chart structure and templates. Reports [ERROR] for install failures and [WARNING] for convention violations.

    helm lint ./mychart -f prod-values.yaml

    Integrate into CI.

    helm template

    Renders all templates locally without a cluster:

    helm template myapp ./mychart --show-only templates/deployment.yaml
helm template myapp ./mychart --set replicaCount=3
helm template myapp ./mychart --debug

    Your primary debugging tool — run constantly while developing.

    helm install --dry-run

    Requires a live cluster. Renders templates AND sends to the K8s API for validation without creating resources:

    helm install --dry-run=server myapp ./mychart

    Use --dry-run=server (Helm 3.13+) for server-side validation. Catches invalid fields and wrong API versions.

    Recommended workflow

    Progressive validation:

    helm lint ./mychart
helm template myapp ./mychart
helm template myapp ./mychart | kubectl apply --dry-run=client -f -
helm install --dry-run myapp ./mychart

    In CI, run all four.

    Gotcha

    helm template cannot evaluate lookup functions (no cluster) — they return empty. --dry-run does NOT execute hooks or wait for readiness. A passing dry-run is necessary but not sufficient for catching all runtime issues.

    Packaging — helm package and .helmignore

    helm package creates a versioned .tgz archive, and .helmignore controls excluded files.

    helm package

    Produces a versioned archive:

    helm package ./mychart
helm package ./mychart --destination ./dist --dependency-update
helm package ./mychart --sign --key mykey

    The .tgz can be pushed to OCI registries with helm push.

    .helmignore file

    Works like .gitignore — one pattern per line, placed in the chart root. Controls which files are excluded from the .tgz. Supports glob patterns, directory patterns, and negation.

    What to exclude

    Always exclude: .git/, .gitignore, .vscode/, .idea/, *.swp, .DS_Store, CI config, test fixtures, and files with secrets. The goal is a clean, minimal package.

    Gotcha

    .helmignore doesn't support ** (double-star) for recursive matching. Files starting with _ in templates/ are included in the package (they need to be — _helpers.tpl is essential). The .helmignore file itself is never included.

    Chart Hooks

    Hooks run Kubernetes resources at strategic lifecycle points using annotations for timing, ordering, and cleanup.

    Hook types

    Nine annotations for helm.sh/hook: pre-install, post-install, pre-delete, post-delete, pre-upgrade, post-upgrade, pre-rollback, post-rollback, test. A single resource can declare multiple:

    annotations:
  "helm.sh/hook": pre-install,pre-upgrade

    Hook weights and ordering

    Execute in ascending order — negative first:

    annotations:
  "helm.sh/hook-weight": "-5"

    Default weight is 0. Use for sequencing: -10 backup, -5 migration, 0 cache warming.

    Delete policies

    Control hook cleanup:

    annotations:
  "helm.sh/hook-delete-policy": before-hook-creation

    Options: before-hook-creation (default), hook-succeeded, hook-failed. Combine: "hook-succeeded,hook-failed" for always-cleanup. Without a policy, hook resources accumulate.

    Example: migration Job

    A Job with annotations:

    annotations:
  "helm.sh/hook": pre-upgrade
  "helm.sh/hook-weight": "-5"
  "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded

    Runs migration before app pods update, cleans up on success.

    Gotcha

    Hook resources are NOT managed as part of the release — helm uninstall won't delete them without a delete policy. For Jobs and Pods, Helm waits until completion. If a hook Job fails, the entire install/upgrade fails.

  • 13
    Subchart Communication

    Parent charts pass values to subcharts through scoped overrides and global values that flow through the entire tree.

    Value scoping

    Subcharts only see their own scoped values — they cannot access parent values directly. The template engine filters .Values to only the subchart's section from the parent.

    Parent overrides

    In the parent's values.yaml, nest values under the subchart's name:

    postgresql:
  auth:
    postgresPassword: secret

    The subchart accesses them normally as .Values.auth.postgresPassword.

    Global values

    Values under the reserved global: key are accessible from ALL charts using .Values.global.*:

    global:
  imageRegistry: myregistry.io

    Globals flow down the entire dependency tree.

    Shared templates

    Named templates defined in any chart are globally available. A parent can include a subchart's template and vice versa. This is how library charts work. Always prefix template names with chart name to avoid collisions.

    values.schema.json — Input Validation

    A JSON Schema file validates user-supplied values during install, upgrade, lint, and template operations.

    How it works

    Create values.schema.json alongside values.yaml. Helm validates final merged values against this schema during install, upgrade, template, and lint. Failures block the operation with a descriptive error. Uses JSON Schema draft-07.

    Common validations

    Type enforcement, enums, and ranges:

    {
  "replicaCount": { "type": "integer" },
  "pullPolicy": { "enum": ["Always", "IfNotPresent", "Never"] },
  "port": { "minimum": 1, "maximum": 65535 }
}

    Also supports required fields and string patterns for naming conventions.

    Gotcha

    The schema validates MERGED values, not just overrides — your values.yaml defaults must also pass. Helm won't warn about keys missing from the schema. Subchart schemas are validated independently. Keep schema and values.yaml in sync manually — tools like helm-schema can help automate this.

    CRDs in Charts — the crds/ Directory

    The crds/ directory holds Custom Resource Definitions installed before other resources, with significant limitations.

    How it works

    Place CRD YAML in crds/. During helm install, Helm uploads CRDs first, waits for API registration, then renders the rest. Multiple CRDs can share a file separated by ---.

    Critical limitations

    CRDs in crds/ are NOT templated — plain YAML only. They are NEVER upgraded or deleted by Helm — only installed on first helm install. Subsequent upgrades skip crds/ entirely. Use --skip-crds to skip installation.

    Best practices

    For simple, stable CRDs: use crds/. For CRDs needing templating or lifecycle management: put them in templates/ instead. For shared CRDs: create a separate dedicated CRD chart. Many projects (cert-manager, Prometheus) provide separate CRD charts for this reason.

    Gotcha

    Since Helm won't upgrade CRDs, use:

    kubectl apply -f crds/

    manually. This is deliberate — CRDs are cluster-scoped and automatic upgrades could break other releases. The separate CRD chart approach avoids these limitations.

    Common Patterns — Labels, Naming, ConfigMaps

    Standard patterns for resource labeling, naming, service accounts, and ConfigMaps make charts consistent and maintainable.

    Standard labels

    Every resource should have: app.kubernetes.io/name, app.kubernetes.io/instance, app.kubernetes.io/version, app.kubernetes.io/managed-by, helm.sh/chart. Define in _helpers.tpl as a named template and include everywhere.

    Selector labels

    Pod selectors must be IMMUTABLE after creation. Use only name + instance for selectors — never version or chart labels that change on upgrade. Define a separate selectorLabels template. Getting this wrong causes "field is immutable" errors.

    Resource naming with fullname

    The fullname helper generates a unique name combining release and chart name, truncated to 63 characters. It supports nameOverride and fullnameOverride. Always use fullname — hardcoded names cause collisions with multiple releases.

    ConfigMaps and auto-rollout

    Create ConfigMaps from files and trigger pod restart on config changes:

    data:
  {{ (.Files.Glob "config/*").AsConfig | nindent 2 }}

    Add a pod annotation to force rolling updates:

    checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
